Project Name  : AI DevOps Copilot
Location      : C:\Users\rahul\Desktop\code\ai-devops-copilot
Type          : Microservices Monorepo
Goal          : A self-hosted AI-powered DevOps assistant that:
                  - Monitors server/cloud infrastructure metrics (CPU, RAM, Disk)
                  - Uses an AI/LLM engine to analyze those metrics
                  - Suggests or automatically executes corrective actions

detects anomalies (like high CPU spikes or disk filling up), and tells
you what to do about it, or does it automatically.

3. ARCHITECTURE — HOW SERVICES TALK TO EACH OTHER
================================================================================

Current Port Assignments:
  - PostgreSQL      : 5432  (Docker internal + host exposed)
  - auth-service    : 3001  (Node.js REST API)
  - metrics-service : 8001  (Python FastAPI REST API)
  - ai-engine       : 8000  (Python FastAPI REST API — placeholder)

Request Flow (what is built so far):

  [User / Client]
       |
       |-- POST /auth/register   --> auth-service:3001
       |-- POST /auth/login      --> auth-service:3001 (returns JWT token)
       |
       |-- GET /metrics/latest   --> metrics-service:8001
       |                              |
       |                              |-- Internally calls: auth-service:3001/auth/me
       |                              |    (to validate the Bearer token)
       |                              |
       |                              `-- Queries PostgreSQL (metricsdb)
       |
       `-- POST /metrics/collect --> metrics-service:8001 (manual trigger)

Scheduled Background Collection (every 60 seconds, automatic):
  metrics-service scheduler
       |
       `-- local_collector.py (reads CPU/RAM/Disk via psutil)
            |
            `-- normalizer.py (formats data)
                 |
                 `-- Saves row to PostgreSQL metricsdb.metrics table

Docker Network:
  All services share "copilot-net" bridge network.
  Services reference each other by container name (e.g., "postgres", "auth-service").

4. SERVICE 1: auth-service (Node.js / Express)
================================================================================

Language   : JavaScript (Node.js 20)
Framework  : Express 4.x
Database   : PostgreSQL (via 'pg' driver, connection pool)
Port       : 3001
Dockerfile : FROM node:20-alpine

Dependencies:
  - express         ^4.19.2  — HTTP server
  - bcrypt          ^5.1.1   — Password hashing (12 salt rounds)
  - jsonwebtoken    ^9.0.2   — JWT creation & verification
  - pg              ^8.11.5  — PostgreSQL client
  - dotenv          ^16.4.5  — Load .env variables
  - validator       ^13.12.0 — Email format validation
  - nodemon         ^3.1.0   — Dev: auto-restart on file change

Database Table (auto-created on startup):
  CREATE TABLE users (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    email         VARCHAR(255) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    created_at    TIMESTAMP WITH TIME ZONE DEFAULT NOW()
  );

API Endpoints:
  POST /auth/register
    Body: { email, password }
    - Validates email format, password length >= 8
    - Checks for duplicate email (409 if exists)
    - Hashes password with bcrypt (12 rounds)
    - Returns JWT token + user info (201)

  POST /auth/login
    Body: { email, password }
    - Finds user by email
    - Compares provided password with stored hash
    - Returns JWT token + user info (200)

  GET /auth/me   [PROTECTED — requires Bearer token]
    - Validates JWT from Authorization header
    - Returns current user's profile (id, email, created_at)

  GET /auth/health
    - Returns { status: "ok" }
    - Used by metrics-service to check auth-service is alive

JWT Token:
  - Contains: { userId, email }
  - Expires: 7 days
  - Secret: from env variable JWT_SECRET

Startup Sequence:
  1. Load env variables (dotenv)
  2. Try to connect to PostgreSQL
  3. CREATE TABLE users IF NOT EXISTS
  4. Start Express server on PORT

5. SERVICE 2: metrics-service (Python / FastAPI)
================================================================================

Language   : Python 3.11
Framework  : FastAPI + Uvicorn
Database   : PostgreSQL (via SQLAlchemy async + asyncpg driver)
Port       : 8001
Dockerfile : FROM python:3.11-slim

Key Dependencies:
  - fastapi==0.111.0        — REST API framework
  - uvicorn[standard]       — ASGI server
  - sqlalchemy[asyncio]     — ORM (async)
  - asyncpg==0.29.0         — Async PostgreSQL driver
  - psutil==5.9.8           — Read CPU/RAM/Disk from OS
  - apscheduler==3.10.4     — Background job scheduler
  - httpx==0.27.0           — HTTP client (to call auth-service)
  - pydantic==2.7.1         — Data validation + response schemas
  - pydantic-settings        — Load .env into Settings class
  - boto3==1.34.120         — AWS SDK (for future CloudWatch collector)

Database Table (auto-created via SQLAlchemy on startup):
  metrics (
    id           UUID PRIMARY KEY,
    user_id      UUID NOT NULL,
    service_name VARCHAR(100) NOT NULL,
    cpu_percent  FLOAT NOT NULL,
    ram_percent  FLOAT NOT NULL,
    disk_percent FLOAT NOT NULL,
    source       VARCHAR(50) DEFAULT 'local',
    region       VARCHAR(50) DEFAULT 'local',
    timestamp    TIMESTAMP WITH TIME ZONE
  )
  Indexes: (user_id, service_name), (timestamp)

API Endpoints:
  GET /metrics/health
    - Returns scheduler status (running/stopped) + "ok"

  GET /metrics/latest?user_id=...&service_name=...   [PROTECTED]
    - Returns the most recent metric row for the given user + service

  GET /metrics/history?user_id=...&service_name=...&hours=1   [PROTECTED]
    - Returns all metric rows within the last N hours (1–24)

  GET /metrics/services?user_id=...   [PROTECTED]
    - Returns a distinct list of service_names being monitored for a user

  POST /metrics/collect   [PROTECTED]
    - Manually triggers one collection cycle immediately
    - Saves result to DB and returns the saved metric

Authentication Flow for Protected Endpoints:
  - Client sends "Authorization: Bearer <token>" header
  - metrics-service calls GET http://auth-service:3001/auth/me
  - If auth-service returns 200, the request proceeds
  - If auth-service is unreachable, returns 503

Collectors:
  local_collector.py (ACTIVE)
    - Uses psutil.cpu_percent(interval=1) — 1-second sampling for accuracy
    - Uses psutil.virtual_memory().percent
    - Uses psutil.disk_usage("/").percent
    - Tags reading as user_id=DEFAULT_USER_ID, service_name="local-machine"

  aws_collector.py (STUB — NOT IMPLEMENTED)
    - Skeleton class AWSCloudWatchCollector exists
    - All methods raise NotImplementedError
    - Planned for Phase 3 to pull EC2/ECS/RDS metrics from AWS CloudWatch

Background Scheduler:
  - Runs on startup via APScheduler (AsyncIOScheduler)
  - Default interval: 60 seconds (configurable via COLLECTION_INTERVAL_SECONDS env)
  - Collects -> normalizes -> saves to DB automatically
  - Errors are caught and logged — a single failure never crashes the scheduler

Normalizer:
  - Validates raw collector output has required fields
  - Rounds float values to 2 decimal places
  - Fills in defaults for optional fields (source, region, timestamp)

CRITICAL ISSUES:
  1. JWT_SECRET is still default value in auth-service/.env
     Current: "your-super-secret-jwt-key-change-in-production-min-32-chars"
     Fix    : Generate a real random secret (e.g., openssl rand -hex 32)

  2. ai-engine is a non-functional placeholder
     It is in docker-compose but does nothing meaningful.
     Users/services that depend on it will get no useful response.

  3. metricsdb authentication issue
     The docker-compose creates auth-service with POSTGRES_DB=authdb,
     but metricsdb is created by init.sql.
     The PostgreSQL user "authuser" is set up for authdb by default.
     VERIFY: Does authuser have CONNECT and CREATE permissions on metricsdb?
     init.sql only creates the database, it does NOT grant permissions explicitly.
     This could cause metrics-service to fail connecting to metricsdb.

MISSING CONFIGURATIONS:
  4. ai-engine has no .env file / .env.example
     Once AI is integrated, it will need: GEMINI_API_KEY (or OPENAI_API_KEY)

  5. No root .gitignore
     Risk: node_modules, .env files, __pycache__ etc. could be committed to git

  6. README.md is empty
     Anyone cloning the repo has no idea how to set it up

  7. No root package.json
     Cannot run all services from the root directory with a single command

ARCHITECTURE GAPS:
  8. action-service does not exist — ai-engine has nowhere to send action triggers
  9. alert-service does not exist — no notification when anomalies happen
  10. frontend does not exist — no user interface at all
  11. ai-engine is not connected to metrics-service in any way
  12. If metrics-service collects data, no one is analyzing or acting on it

CODE ISSUES:
  13. auth-service/.env — DB_PASSWORD is hardcoded as "authpassword123"
      This matches docker-compose but is insecure for production use.

  14. metrics-service DEFAULT_USER_ID is a hardcoded UUID
      "00000000-0000-0000-0000-000000000001"
      This only works for single-user testing. For real multi-user use,
      the scheduler needs to know which authenticated user to tag metrics under.

  15. local_collector.py uses disk_usage("/") — this is "/" (Linux root)
      This works inside Docker container but the "/" inside Docker is the
      container's filesystem, not the real host disk. This reading will not
      reflect actual host disk usage accurately.

  16. ai-engine Dockerfile uses "FROM python:3.10" (full image, not slim)
      Should use "FROM python:3.10-slim" like metrics-service for smaller image.

  17. docker-compose.yml has no env_file references for auth-service or metrics-service
      Environment variables are hardcoded directly in docker-compose.yml
      This is fine for dev but environment values should ideally be loaded from .env files.
