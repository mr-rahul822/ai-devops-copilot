# AI DevOps Copilot - Deep Analytical Project Report

## Introduction
This report provides a deep analytical overview of the "AI DevOps Copilot" project. It is intended to serve as comprehensive context for Claude or any other AI/Developer to quickly understand the architecture, routes, endpoints, data flow, and current state of the application. 

**Perspective**: Senior Tester & Senior DevOps Engineer.

---

## 1. High-Level Architecture
The project is a distributed microservices architecture, containerised using Docker Compose.

**Tech Stack**:
- **Frontend**: React, Vite, TailwindCSS, Zustand (State Management), React Query, Axios.
- **Backend Services**: Mixture of Node.js (Express) and Python (FastAPI).
- **Databases & Stores**: PostgreSQL (Relational Data), Pinecone (Vector Database for RAG).
- **Message Broker**: Kafka (with Zookeeper) for asynchronous metric streaming.
- **AI Integration**: Langchain/Custom LLM clients wrapping Gemini/Claude APIs, HuggingFace embeddings (`all-MiniLM-L6-v2`).

---

## 2. Microservices Breakdown & Endpoints

### 2.1. `auth-service` (Node.js / Express - Port: 3001)
Handles user authentication, JWT lifecycle, sessions, and MFA.
- **Datastore**: PostgreSQL (`authdb`), `users`, `refresh_tokens`, `active_sessions`, `login_audit_log` tables.
- **Endpoints**:
  - `POST /auth/register` - Registers a new user.
  - `POST /auth/login` - Authenticates user, issues access (JWT) and refresh tokens.
  - `POST /auth/refresh` - Issues new access token via refresh token cookie.
  - `GET /auth/me` - Validates JWT and returns current user info.
  - `POST /auth/logout` - Revokes tokens and clears cookies.
  - `GET /auth/health` - Basic healthcheck.
  - Profile Sub-routes: `GET/PATCH /auth/profile`, `POST /auth/change-password`, `POST /auth/reset-password`.
  - MFA Sub-routes: `POST /auth/mfa/setup`, `POST /auth/mfa/verify-setup`, `POST /auth/mfa/validate`, `POST /auth/mfa/disable`.
  - Sessions Sub-routes: `GET /auth/sessions`, `DELETE /auth/sessions/:id`, `DELETE /auth/sessions/all`.
  - Audit Sub-route: `GET /auth/audit-log`.

### 2.2. `metrics-service` (Python / FastAPI - Port: 8001)
Collects metrics (CPU, RAM, Disk) from local containers and AWS CloudWatch, saves them to DB, and publishes them to Kafka.
- **Datastore**: PostgreSQL (`metricsdb`), `metrics` table.
- **Message Broker**: Publishes to Kafka topic `metrics-stream`.
- **Endpoints**:
  - `GET /metrics/health` - Service health and APScheduler status.
  - `GET /metrics/latest` - Most recent metrics across services.
  - `GET /metrics/history` (Alias `GET /metrics`) - Historical metrics with time filters.
  - `GET /metrics/summary` - Aggregated stats (avg CPU/RAM) for dashboard cards.
  - `GET /metrics/services` - Distinct list of monitored services.
  - `POST /metrics/collect` - Manual trigger for metric collection loop.
  - `POST /cloud/connect`, `GET /cloud/status`, `DELETE /cloud/disconnect`, `GET /cloud/instances` - AWS CloudWatch onboarding.

### 2.3. `alert-service` (Node.js / Express - Port: 3003)
Consumes Kafka metrics, evaluates them against threshold rules, and generates alerts.
- **Datastore**: PostgreSQL (`alertsdb`), `alerts` table.
- **Message Broker**: Consumes from Kafka topic `metrics-stream`.
- **Alert Engine**: Evaluates consecutive breaches (e.g., CPU > 85% for 3 readings) and a `SERVICE_SILENT` cron job.
- **Endpoints**:
  - `GET /alerts/health` - Returns Kafka consumer status.
  - `GET /alerts/summary` - Dashboard badge counts (open/critical).
  - `GET /alerts` - Lists alerts with filtering.
  - `GET /alerts/:id` - Single alert details.
  - `POST /alerts/:id/resolve` - Marks alert as resolved.

### 2.4. `ai-engine` (Python / FastAPI - Port: 8002)
The brain of the platform. Analyzes logs/metrics, provides chat, and resolves incidents via multi-agent pipelines and RAG.
- **Datastore**: PostgreSQL (`authdb` -> `incidents` table), Pinecone (Vector Index: `devops-incidents`).
- **AI Pipelines**: Uses `LogAnalyzerAgent`, `MetricsAnalyzerAgent`, `DecisionAgent`, `ExecutorAgent`.
- **Endpoints**:
  - `GET /ai/health` - Checks Pinecone and LLM connectivity.
  - `POST /ai/analyze` - Runs the 4-agent pipeline on logs and metrics to find root causes and propose action plans.
  - `POST /ai/diagnose` - End-to-end diagnosis pipeline fetching metrics/alerts to generate AI diagnosis.
  - `POST /ai/chat` - Context-aware conversational endpoint.
  - `GET /ai/incidents` - Lists past AI-diagnosed incidents.
  - `GET /ai/incidents/{incident_id}` - Incident details.
  - `POST /ai/incidents/{incident_id}/resolve` - Marks incident as resolved and **embeds it into Pinecone** for future RAG.

### 2.5. `action-service` (Python / FastAPI - Port: 8003)
Executes AI-recommended infrastructure fixes via Docker or AWS APIs with an approval gate and audit logging.
- **Datastore**: PostgreSQL (`actionsdb`), `actions` and `audit_logs` tables.
- **Executors**: `docker_executor.py` (Local Docker Socket) and `aws_executor.py` (Boto3).
- **Endpoints**:
  - `GET /actions/health` - Checks Docker and AWS connectivity.
  - `POST /actions/execute` - Requests an action. If low-risk, auto-executes; else goes to `PENDING_APPROVAL`.
  - `POST /actions/{action_id}/approve` - Approves and executes a pending action.
  - `POST /actions/{action_id}/reject` - Rejects an action.
  - `GET /actions` - Lists actions.
  - `GET /actions/{action_id}` - Action details + full audit trail.
  - `GET /actions/{action_id}/logs` - Returns logs from `fetch_container_logs` actions.

### 2.6. `frontend` (React / Vite - Port: 5173 / 80 in Docker)
- **State Management**: Zustand (`useStore`).
- **API Client**: Axios instances configured with intercepters for auto-refreshing JWTs and attaching CSRF tokens.
- **Pages**: Dashboard, Alerts, LogExplorer, Chat, Actions, Settings, CloudConfiguration, AIInsights, Profile, Login.

## 3. Data Flow & Event Driven Architecture

1. **Metrics Collection**: `metrics-service`'s APScheduler runs every 60s. It fetches AWS CW data and local stats, saves to PostgreSQL, and pushes to Kafka `metrics-stream`.
2. **Alert Evaluation**: `alert-service` consumes Kafka. If CPU > 85% 3 times, it triggers an alert and saves it to DB.
3. **AI Diagnosis**: User sees alert on Frontend, clicks "Diagnose". Calls `/ai/analyze` or `/ai/diagnose`.
4. **Multi-Agent Pipeline**: 
   - Agent 1 parses logs via Regex.
   - Agent 2 analyzes metric trends.
   - Agent 3 (LLM) queries Pinecone RAG for past resolved incidents, formulates root cause and recommended actions.
   - Agent 4 creates an Action Plan.
5. **Action Execution**: User clicks "Apply Fix". Frontend calls `/actions/execute`.
6. **Approval Gate**: If High Risk, action pends. User approves via `/actions/:id/approve`.
7. **Resolution Loop**: `action-service` runs the Docker/AWS command. User marks incident resolved via `/ai/incidents/:id/resolve`, which embeds the fix into Pinecone to make the AI smarter next time.

## 4. DevOps & QA Analysis: What's Missing / Tech Debt

As a Senior DevOps Engineer and Senior Tester, here are the critical gaps and areas for improvement in the current state of the repository:

### 4.1. Testing & QA Gaps
- **Lack of Unit/Integration Tests**: There are no `__tests__` or `tests/` directories for Jest, Pytest, Mocha, or Mocha. The `test-tools` directory contains manual Python scripts (`test_phase6.py`, etc.) for seeding/E2E emulation, not automated CI-ready test suites.
- **No E2E UI Testing**: Missing Cypress, Playwright, or Selenium setups for frontend testing.
- **No Load/Stress Testing**: Given Kafka and microservices, missing k6, Locust, or JMeter scripts to test the event-driven bottlenecks.

### 4.2. DevOps & Infrastructure Gaps
- **Missing CI/CD Pipelines**: No `.github/workflows`, `.gitlab-ci.yml`, or Jenkinsfiles. Build, linting, testing, and Docker push processes are completely manual.
- **Hardcoded Secrets & Environment Vars**: The `docker-compose.yml` file contains hardcoded `POSTGRES_PASSWORD`, `JWT_SECRET`, and database connection strings. This is a massive security risk. These must be moved to `.env` files entirely or a Secrets Manager.
- **Missing Infrastructure as Code (IaC)**: Reliance purely on `docker-compose.yml` is fine for local dev, but missing Terraform (`.tf`), Pulumi, or Ansible for provisioning the actual AWS EC2/RDS infrastructure.
- **No Kubernetes Configs**: Lacks Helm charts or raw YAML manifests (`Deployments`, `Services`, `Ingress`) for production-grade orchestration.
- **Logging & Observability for the Tool Itself**: Ironically for a DevOps monitoring tool, the microservices themselves lack centralized logging (ELK, Loki) and Prometheus metrics endpoints. They just log to `stdout`.

### 4.3. Application Logic / Codebase Smells
- **Mocked AWS Metrics**: `aws_collector.py` hardcodes `ram_percent = 40.0` and `disk_percent = 25.0` because CloudWatch doesn't provide these natively without the CW Agent.
- **LLM Client Naming**: `client.py` in `ai-engine` is named `ClaudeClient`, but is hardcoded to call the Gemini API (`gemini-2.5-flash`) via an OpenAI compatibility layer. This is highly confusing.
- **Missing Swagger/OpenAPI for Node.js**: While FastAPI auto-generates `/docs`, the Express services (`auth-service`, `alert-service`) lack Swagger/OpenAPI documentation.
- **Kafka Initialization**: The `docker-compose.yml` uses a hacky `bash -c "cub kafka-ready ..."` container to create topics. A more robust initialization pattern or native Kafka auto-create should be considered.
- **CORS Configuration**: CORS is overly permissive (`allow_origins=["*"]`) in FastAPI services, which is unsafe for production.

---

## Conclusion
The AI DevOps Copilot is a highly functional, well-architected prototype demonstrating complex patterns (Event-Driven Architecture, Microservices, RAG, Multi-Agent LLMs). 

However, to transition from a "Production-Ready Prototype" to an actual "Enterprise Deployment", it desperately needs Automated Testing (Pytest/Jest), CI/CD pipelines, proper Secrets Management, and Infrastructure as Code (Terraform/Kubernetes).
